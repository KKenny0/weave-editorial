# Weave Editorial

把研究成稿变成一篇读者愿意读下去、又不牺牲内容完整性的跨平台文章。

## 结果契约

输入是一篇已经完成研究和引用核验的中文 Markdown 长文。输出是一篇完整 Markdown 成稿，可同时用于微信公众号和 X 的长文发布形态。

只改变：

- 标题与开头；
- 信息出现顺序；
- 主叙事、次叙事、证据和边界说明的强弱；
- 段落衔接、小标题与结尾；
- 为可读性所需的句式，但不改变事实含义。

保留源稿中的每一项独立信息、限定条件、反例、来源、图片、表格和代码块。次要材料可以后移、并入较弱位置或缩短过渡，不能因“提高传播力”而删除。

## 使用边界

使用本 skill 前，确认：

1. 输入是单篇 Markdown 文件，而不是资料包。
2. 文章已经完成研究和事实核验。
3. 用户要的是一篇完整成稿，而不是多个平台版本。

如果需要补研究、补来源或核验事实，先交给研究流程；如果只需句子润色、翻译、摘要、thread、短帖或卡片，不使用本 skill。

## 开始前

完整阅读输入文件，并读取 [references/editorial-protocol.md](references/editorial-protocol.md)。不要边读边改。

默认输出到源文件同目录，文件名为 `{source-stem}-editorial.md`。用户指定路径时遵从用户。不得覆盖源文件；目标文件已存在时先询问，或选择带序号的新文件名并明确说明。

## 工作流

### 1. 建立内容账本

在内部记录源稿中的内容单元：核心判断、独立事实、例子、反例、限定条件、数字、引用、URL、图片、表格、代码块和覆盖度说明。账本用于防止结构重写时把材料遗落，不写入最终文章。

### 2. 选出唯一主叙事

用一句问题定义读者为什么要读这篇文章。主叙事必须能容纳源稿的核心判断，不得凭空制造更强的结论。

将内容单元分为四类：

- 主叙事：直接推动核心问题；
- 次叙事：有独立价值，但不应打断主线；
- 证据：支撑判断的论文、产品、数据、例子和反例；
- 边界：适用范围、缺口、覆盖度、未竟问题和来源说明。

分类只决定出现位置与篇幅力度，不决定删留。

### 3. 重写标题和开头

标题给出具体问题、冲突或判断，不用夸张承诺。开头在较短篇幅内完成三件事：提出读者问题、说明为什么值得关心、给出全文将回答什么。

不要用行业背景、来源收集过程、空泛趋势或“大而全”定义开场。

### 4. 重排正文

先排主叙事，再把次叙事放入不打断主线的位置。常用弱化方式：

- 放到主线之后；
- 并入相关主章节的后半段；
- 作为“还有一个问题”“这也带来另一层约束”等次级转折；
- 放入结尾前的边界与未竟部分。

同一证据只在最能推动论证的位置完整解释，其他位置用回指连接。不要把文章改成列表摘要。

### 5. 做跨平台兼容

文章必须脱离目录、脚注悬浮、交互组件和平台专属排版也能成立。使用清楚的小标题、适中的段落和自解释的链接文本。保留 Markdown 图片、链接、表格和代码块，不生成“公众号版”“X 版”两个分支。

### 6. 收束结尾

结尾回答开头的问题，给出读者可带走的判断，并保留源稿已有的边界、未竟问题或入门材料。不要新增口号式号召、关注引导或未经源稿支持的预测。

### 7. 验证后交付

写完后逐项对照内容账本，并运行：

```powershell
pwsh -File scripts/check.ps1 -SourceArticle <源稿> -OutputArticle <成稿>
```

如果当前目录不是 skill 根目录，使用本 skill 所在目录下的 `scripts/check.ps1`。修复所有失败项；长度告警需要人工复核，但不能仅为提高比例而堆回重复文字。

最终只交付一个 Markdown 文件，并简要报告输出路径、保真检查结果和任何无法自动判断的风险。

## 硬边界

- 不新增研究、来源、数字、案例或事实判断。
- 不删除独立信息；允许删除纯重复、目录性提示和因重排产生的重复过渡。
- 不改变引文归属、证据强度、时间范围、适用条件和不确定性。
- 不输出摘要、thread、短帖、卡片、发布文案或平台专属副本。
- 不覆盖源文件，不自动发布。
