# 为什么长上下文不能替代记忆

## 一个常见判断

模型上下文越来越长，于是有人认为 Agent 不再需要独立记忆：把历史全部放进上下文即可。这个判断把“可放入”与“应该读取”混在了一起。

![上下文与记忆](https://example.com/images/context-memory.png)

## 两种机制回答不同问题

长上下文扩大单次推理可见的材料。记忆系统决定哪些过去信息应当继续存在、属于谁、何时进入当前任务，以及何时失效。[Context Study](https://example.com/context-study) 测量了长度增加后的检索变化；[Memory Study](https://example.com/memory-study) 则测量了记录更新。

## 注意力不是权限

即使模型能在十万 token 中找到一句话，这句话也可能来自另一个用户或已撤回的任务。注意力权重不能替代访问控制，也不能证明记录仍有效。

## 反例：短任务

对一次性的代码解释任务，完整历史只有几千 token，没有跨用户信息，也不会在未来复用。此时独立记忆会引入额外抽取和维护成本，长上下文已经足够。这个反例说明独立记忆不是每个 Agent 的必选组件。

## 反例：原始记录有审计价值

摘要式记忆会丢失措辞与顺序。如果法律审阅需要看到原始对话，压缩后的记忆不能替代日志。合理设计是保留审计日志，同时让工作记忆只存任务所需的派生记录。

## 方法限制

两项研究使用不同模型、不同任务和不同上下文长度，不能把准确率直接横向比较。Context Study 只运行了两周，没有观察记录过期；Memory Study 没有报告存储成本。

## 维护才是分界线

当历史需要跨任务复用、受权限约束、会过期或可被撤回时，系统必须维护信息生命周期。长上下文解决容量问题，记忆解决生命周期问题。

## 还有一层成本

记忆并不免费。抽取、冲突检测、检索和删除传播都会消耗计算。是否采用独立记忆，应由复用收益是否超过维护成本决定，而不是由上下文窗口长度单独决定。

## 结论

“长上下文还是记忆”不是二选一。短期可见性由上下文承担，跨任务生命周期由记忆系统承担。任务没有长期状态时，后者可以省略。
