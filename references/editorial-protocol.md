# 编辑协议

本协议处理一个特定矛盾：文章需要更容易进入，但不能靠删掉次要内容来获得速度。解决方式是改变叙事层级，而不是减少信息集合。

## 1. 内容账本

完整阅读源稿后，在内部列出以下内容单元：

| 类型 | 记录什么 | 保真要求 |
|---|---|---|
| 判断 | 作者明确提出的结论 | 不提高确定性，不扩大适用范围 |
| 事实 | 数字、时间、机制、产品或论文描述 | 保留限定词与归属 |
| 证据 | 引用、链接、图、表、代码、案例 | 不换来源，不改证据强度 |
| 反证 | 失败案例、反例、相反解释 | 不因影响流畅度而删除 |
| 边界 | 覆盖度、缺口、未竟问题、适用条件 | 保留可核验事实，弱化自我评分 |
| 导航 | 目录、重复预告、章节回顾 | 可随结构重排而重写或删除 |

“独立信息”指删除后，读者会少知道一个事实、条件、例子、反例、问题或来源。纯重复表达不是独立信息。

## 2. 叙事层级

### 主叙事

全文只能有一个主叙事。它通常是一个读者问题，或者一组能被同一判断解释的具体问题。判断标准不是“最宏大”，而是能否串起最多核心证据。

### 次叙事

次叙事有独立价值，但不承担文章的推进节奏。降低它的叙事权重，不等于删减：

1. 先让主叙事形成完整链条；
2. 把次叙事放到相关章节后半段、主线之后或边界部分；
3. 减少铺垫和重复解释，保留其事实、证据和结论；
4. 用一句转折说明它与主线的关系。

不要建立多个并列、同等响亮的“主线”。那会把综述重新变回资料清单。

### 证据与边界

证据紧跟它支撑的判断。边界说明尽量集中，避免不断打断正文；但与某个判断直接相关的限定条件必须就近出现。

## 3. 标题与开头

标题优先回答“读者正在困惑什么”。可用问题、具体冲突或有边界的判断。避免“终极”“彻底”“必读”“颠覆”等承诺，也不要只列领域名加“全景综述”。

开头建议在 3—6 段内完成：

1. 给出一个具体场景或问题；
2. 指明常见理解缺少哪一层；
3. 给出文章的核心观察与阅读地图。

不要从行业规模、竞争格局、资料搜集过程或通用定义开始。研究范围、样本数和覆盖缺口放到文末的覆盖度说明。

## 4. 正文重排

先写出只包含章节名的一条逻辑链，确认每一节都回答前一节留下的问题。然后把账本中的内容单元逐一放回。

优先使用读者会直接提出的问题作为小标题。技术术语首次出现时给出简短中文解释。用具体动词说明论文或系统做了什么，不用“照亮”“撬动”“升维”等编辑性比喻替代事实。

需要压缩时，按以下顺序处理：

1. 删除重复预告、重复回顾和目录式句子；
2. 合并承担相同作用的段落；
3. 缩短背景和过渡；
4. 将旁支移到次级位置；
5. 停止。不得继续删除独立材料。

## 5. 跨平台兼容

成稿需要在没有平台专属组件时仍然成立：

- 只保留一个 H1 标题；
- 小标题能独立说明本节问题；
- 段落不过度依赖连续超长句；
- 链接文本说明链接内容，不使用“点这里”；
- 图片保留原始 Markdown 引用和说明；
- 表格与代码块保持语义完整；
- 不使用“公众号版”“X 版”“第 1 条/共 12 条”等分版标记；
- 不添加关注、转发、评论诱导。

这里的“适配 X”指适配 X 的长文发布形态，不是把文章拆成 thread。

## 6. 语气与事实

- 用事实承重，删除判断性形容词。
- 保留克制的个人声音，例如“我更关心的是”，但不制造强烈立场。
- 以对象本身为中心，不用竞品比较搭建戏剧性。
- 能用具体列表说明时，不使用“覆盖较薄”“能力很强”等抽象判断。
- 不把来源集合的限制伪装成领域结论；覆盖度说明只写可核验事实。

若源稿中的个人语气、厂商名或比较本身就是事实内容，则保留事实，不机械改写。

## 7. 结尾

结尾回到开头的问题，依次给出：

1. 主叙事得出的核心判断；
2. 读者面对类似问题时可以采用的观察框架；
3. 源稿已有的未竟问题、覆盖度与入门材料。

不要在结尾突然扩大结论，也不要用口号代替边界。

## 8. 交付前人工复核

自动检查通过后，再回答以下问题：

- 开头是否在六段内进入核心问题？
- 是否只有一个主叙事？
- 每个次叙事是否仍完整存在，但没有抢走主线？
- 是否有任何事实的确定性、范围或归属被改变？
- 所有图片、链接、表格、代码块和反例是否仍在？
- 结尾是否回答了开头，而不是只做摘要？
- 全文是否是一篇文章，而不是摘要、提纲或 thread？
